1. La App Spring Boot Como Un Restaurante

Una aplicación Spring Boot suele dividirse en capas. Esa división ayuda a que cada parte tenga una responsabilidad clara. En tu proyecto de Empresa se ve así:

Controller
recibe HTTP
→
Service
decide reglas
→
Repository
habla con DB
Analogía: piensa en un restaurante. El Controller es el mozo que recibe el pedido, el Service es el chef que decide cómo prepararlo, y el Repository es la despensa donde se guardan o buscan los ingredientes.
Capa En tu proyecto Responsabilidad Qué se prueba
Controller EmpresaController Recibir requests HTTP y devolver responses. Rutas, status HTTP, estructura JSON.
Service EmpresaServiceImpl Aplicar reglas de negocio. Decisiones: crear, actualizar, listar, eliminar.
Repository EmpresaRepository Consultar o guardar datos. En unit test normalmente se reemplaza por un mock.

Flujo Real Al Crear Empresa

  1. El usuario manda un POST /empresa/v1/.
  2. El EmpresaController recibe el EmpresaRequest.
  3. El controller llama a empresaService.crear(request).
  4. El service pregunta al repository si existe el documento.
  5. Si existe, responde conflicto. Si no existe, guarda y responde OK/CREATED.

2. Qué Es Unit Testing

Un unit test prueba una unidad pequeña del sistema. No intenta levantar todo el mundo. Su misión es responder: “si esta pieza recibe estas condiciones, ¿devuelve el resultado esperado?”.

Unit test bueno

Prueba una clase o método con dependencias controladas.

Ejemplo: probar EmpresaServiceImpl.crear() usando un EmpresaRepository falso.

Unit test confuso

Depende de base de datos, red, servidores externos o datos cambiantes.

Si falla, no sabes si falló tu lógica, PostgreSQL, Docker o la configuración.

Analogía: si quieres probar si una licuadora funciona, no necesitas abrir un restaurante completo. Pones fruta, presionas el botón y miras si salió jugo. Eso es unit testing: aislar lo importante.

Qué Está Probando Tu Test De Service

En EmpresaServiceImplTest estás probando reglas como:

  • Si la empresa ya existe, no debe guardarse otra vez.
  • Si la empresa no existe, debe guardarse correctamente.
  • Si busco un ID existente, debe retornar la empresa.
  • Si busco un ID inexistente, debe retornar “No hay datos”.
  • Si actualizo una empresa inexistente, debe responder que no existe.
  • Si elimino una empresa existente, debe hacer eliminación lógica.
Idea clave: no estás probando Hibernate ni PostgreSQL. Estás probando las decisiones de tu clase de servicio.

3. Mockito: Actores Para Tu Prueba

Mockito sirve para crear objetos falsos que simulan comportamientos. Estos objetos se llaman mocks. En vez de usar una base de datos real, haces que el repository responda exactamente lo que necesitas para cada escenario.

Analogía: Mockito es como contratar actores para una escena. El service es real, pero el repository es un actor que dice las líneas que tú le escribes.
Mockito Qué significa Ejemplo mental
@Mock Crea una dependencia falsa. Una despensa falsa.
@InjectMocks Crea la clase real e inyecta los mocks dentro. Chef real usando despensa falsa.
when(...).thenReturn(...) Define qué debe responder el mock. “Si te preguntan X, responde Y”.
any() Acepta cualquier argumento. “No importa qué empresa llegue”.
anyString() Acepta cualquier texto. “No importa qué número de documento llegue”.

Ejemplo De Tu Proyecto

@Mock
private EmpresaRepository empresaRepository;

@InjectMocks
private EmpresaServiceImpl empresaService;

Esto significa: “quiero probar el EmpresaServiceImpl real, pero no quiero una base de datos real. Métele un EmpresaRepository falso”. 

when(empresaRepository.existsByNumeroDocumento(anyString()))
    .thenReturn(true);

Traducción: “cuando el service pregunte si existe una empresa por documento, responde que sí”.

4. Patrón AAA

AAA es una forma simple y ordenada de escribir tests.

Arrange

Preparar el escenario.

Datos falsos, mocks y respuestas esperadas.

Act

Ejecutar la acción.

Llamar al método real que quieres probar.

Assert

Verificar el resultado.

Comparar lo obtenido con lo esperado.

Analogía: quieres probar una alarma. Arrange: cierras la puerta y activas la alarma. Act: abres la puerta. Assert: verificas que sonó.

AAA En Tu Test “Empresa Ya Existe”

@Test
@DisplayName("Deberia retornar conflicto si la empresa ya existe")
void testDeberiaRetornarEmpresaExiste() {
    // Arrange
    when(empresaRepository.existsByNumeroDocumento(anyString()))
        .thenReturn(true);

    // Act
    ResponseEntity<BaseResponse<Empresa>> resultado =
        empresaService.crear(empresaRequest);

    // Assert
    assertNotNull(resultado);
    assertEquals(Constants.CODE_EXIST, resultado.getBody().getCode());
    assertFalse(resultado.getBody().getObjeto().isPresent());
}
Parte Línea mental Qué demuestra
Arrange El repository dice que la empresa ya existe. Preparas el caso negativo.
Act Llamas a empresaService.crear(). Ejecutas la lógica real.
Assert Esperas código de existencia y objeto vacío. Confirmas que el service tomó la decisión correcta.
Regla práctica: cada test debe contar una historia clara. Si necesitas explicar demasiado el test, quizá está probando muchas cosas a la vez.

5. Tests De Controller Con MockMvc

En EmpresaControllerTest ya no estás probando la lógica completa del service. Estás probando la capa web: rutas, JSON, status HTTP y formato de respuesta. 

@WebMvcTest
class EmpresaControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private EmpresaService empresaService;
}

@WebMvcTest levanta una parte pequeña de Spring enfocada en MVC. El controller es real, pero el service se reemplaza por un mock.

Analogía: aquí no estás probando al chef. Estás probando al mozo: si recibe un pedido por la ruta correcta, si lo manda al service y si entrega una respuesta con buen formato.

Ejemplo: POST Para Registrar Empresa

mockMvc.perform(MockMvcRequestBuilders.post("/empresa/v1/")
        .contentType(MediaType.APPLICATION_JSON)
        .content(objectMapper.writeValueAsString(empresaRequest)))
    .andExpect(status().isOk())
    .andExpect(jsonPath("$.code").value(2001))
    .andExpect(jsonPath("$.objeto.numeroDocumento").value("123456789"));
Elemento Qué hace
MockMvc Simula requests HTTP sin abrir un servidor real.
ObjectMapper Convierte el objeto Java a JSON.
jsonPath("$.code") Busca el campo code dentro del JSON de respuesta.
@MockBean Reemplaza un bean de Spring por un mock dentro del contexto de test.

6. Tipos De Pruebas En Spring Boot

Tipo Qué prueba Herramientas comunes Cuándo usarlo
Unit test Una clase aislada. JUnit, Mockito. Reglas de negocio del service.
Web MVC test Controller y capa web. @WebMvcTest, MockMvc. Rutas, JSON, status.
Integration test Varias capas juntas. @SpringBootTest, DB real o Testcontainers. Validar que todo conecta bien.
Repository test Consultas a base de datos. @DataJpaTest. Probar queries, mappings y JPA.
Importante: no todo test debe levantar Spring completo. Mientras más grande el contexto, más lento y más difícil de diagnosticar será el fallo.

7. Principios SOLID En Spring Boot

SOLID son cinco principios para diseñar código más mantenible. No son reglas mágicas, pero sí ayudan a que tu proyecto sea más fácil de probar, cambiar y entender.

Analogía general: SOLID es como organizar una cocina profesional. Cada persona tiene una tarea, las herramientas tienen un uso claro, y puedes cambiar una pieza sin destruir todo el restaurante.

S: Single Responsibility Principle

Una clase debe tener una sola razón para cambiar.

En Spring Boot, el controller no debería contener reglas de negocio ni SQL. El service no debería saber de HTTP puro. El repository no debería decidir mensajes de respuesta.

Mal enfoque Buen enfoque
El controller valida duplicados, guarda en DB y arma toda la lógica. El controller recibe request y delega al service.
El service arma rutas HTTP. El service aplica reglas: existe, no existe, guardar, actualizar.

En tu proyecto, EmpresaController delega en EmpresaService. Eso se acerca bien a SRP.

O: Open/Closed Principle

El código debería estar abierto a extensión, pero cerrado a modificación.

Significa que si mañana agregas una nueva forma de validar empresas, idealmente no deberías romper todo lo existente.

Si tienes una cafetera modular, puedes agregar un filtro nuevo sin reconstruir la cocina.

En Spring Boot esto se logra usando interfaces, clases pequeñas y estrategias. Por ejemplo: 

public interface EmpresaValidator {
    void validar(EmpresaRequest request);
}

Luego podrías tener DocumentoValidator, DireccionValidator o RucValidator sin meter todo dentro de un único método enorme.

L: Liskov Substitution Principle

Una implementación debe poder reemplazar a su abstracción sin romper el programa.

Si una clase promete comportarse como EmpresaService, cualquier implementación debería respetar ese contrato.

Si pides “un cargador USB-C”, cualquier cargador compatible debería funcionar. Si uno tiene forma USB-C pero quema el equipo, no respeta el contrato.

En Spring Boot, si tienes: 

private final EmpresaService service;

El controller no debería depender de detalles internos de EmpresaServiceImpl. Solo necesita que el contrato de EmpresaService se cumpla.

I: Interface Segregation Principle

Una clase no debería estar obligada a implementar métodos que no necesita.

Si una interfaz crece demasiado, las implementaciones terminan con métodos vacíos, falsos o con return null.

En tu service aparece obtenerEmpresaXNumDoc retornando null. Para aprender assertNull está bien, pero en diseño real puede ser señal de contrato incompleto o método pendiente.

Si una interfaz tiene demasiadas responsabilidades, puedes separarla: 

public interface EmpresaQueryService {
    ResponseEntity<BaseResponse<Empresa>> obtenerEmpresa(Long id);
}

public interface EmpresaCommandService {
    ResponseEntity<BaseResponse<Empresa>> crear(EmpresaRequest request);
    ResponseEntity<BaseResponse<Empresa>> actualizar(Long id, EmpresaRequest request);
}

D: Dependency Inversion Principle

Las clases de alto nivel deben depender de abstracciones, no de detalles concretos.

Tu controller depende de EmpresaService, no directamente de EmpresaServiceImpl. Eso permite reemplazar la implementación, probar con mocks y mantener bajo el acoplamiento.

Un conductor no debería depender de una marca específica de motor para manejar. Depende de una interfaz: volante, acelerador, freno. El motor concreto puede cambiar. 
@RestController
@RequiredArgsConstructor
public class EmpresaController {
    private final EmpresaService service;
}

Esta idea también hace posible tus tests de controller: reemplazas EmpresaService por un mock con @MockBean.

8. Cómo Pensar Un Test Nuevo

Antes de escribir código, escribe la historia del test en una frase.

Pregunta Ejemplo
¿Qué quiero probar? Crear empresa cuando no existe.
¿Qué condición preparo? El repository dice que el documento no existe.
¿Qué acción ejecuto? Llamo a empresaService.crear(request).
¿Qué resultado espero? Código OK, objeto presente y llamada a save.

Plantilla Recomendada

@Test
@DisplayName("Deberia hacer X cuando ocurre Y")
void deberiaHacerXCuandoOcurreY() {
    // Arrange
    // datos y mocks

    // Act
    // metodo real

    // Assert
    // verificaciones
}

Checklist Mental

  • ¿El nombre del test cuenta una historia?
  • ¿El Arrange prepara solo lo necesario?
  • ¿El Act tiene una acción principal?
  • ¿El Assert verifica comportamiento, no detalles irrelevantes?
  • ¿El test falla si la lógica real se rompe?
  • ¿El test no depende de PostgreSQL si es unit test?
